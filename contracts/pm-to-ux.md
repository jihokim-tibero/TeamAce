# PM → UX 핸드오프 계약

## PM이 UX에게 전달해야 하는 필수 산출물

### 1. 기획서 (PRD)
- [ ] 배경 및 목적 (왜 이 기능이 필요한가)
- [ ] 타깃 사용자 정의
- [ ] 사용자 시나리오 (주요 흐름 + 예외 흐름)
- [ ] 목표 지표 (측정 가능한 KPI)
- [ ] 범위 (In Scope / Out of Scope)

### 2. 기능정의서
- [ ] 기능 목록 테이블 (기능ID, 기능명, 우선순위)
- [ ] 각 기능의 입력 조건 / 처리 로직 / 출력 결과 / 예외 처리
- [ ] 상태 전이가 있는 기능은 상태 목록 명시

### 3. UX 에이전트를 위한 지시
- [ ] 필요한 화면 목록 (또는 "UX 판단에 위임" 명시)
- [ ] 특별한 UX 제약사항 (있는 경우)
- [ ] 참고할 경쟁사/레퍼런스 (있는 경우)

## 필드 네이밍
- 기능정의서에 정의된 필드명이 이후 전체 파이프라인의 기준
- UX는 이 필드명을 data-testid, 컴포넌트 props에 그대로 사용

## 산출물 위치
- 기획서 (PRD): GitHub → **Notion** (Notion MCP) / GitLab → **Git Wiki** (`glab wiki`)
- 기능정의서: GitHub → **Notion** (Notion MCP) / GitLab → **Git Wiki** (`glab wiki`)
- UX Agent는 Notion 또는 Git Wiki에서 기획서·기능정의서를 조회

## 검증 기준
- UX Agent가 Notion/Wiki의 기획서만 보고 "어떤 화면이 필요한지, 각 화면에서 무슨 일이 일어나는지" 파악 가능해야 함
- 모호한 표현 금지: "적절히 표시" → "성공 시 결과 카드 표시, 실패 시 에러 메시지 + 재시도 버튼 표시"
