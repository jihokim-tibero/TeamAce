# QA → 전체 피드백 계약

## QA가 다른 에이전트에게 전달하는 산출물

### 1. 테스트 결과 리포트
- [ ] TC 총 수, 통과/실패/스킵 수
- [ ] Critical TC 통과율 (100% 필수)
- [ ] 테스트 커버리지 (단위/통합/E2E)

### 2. 버그 리포트 (발견 시)
- [ ] 심각도: Critical / Major / Minor
- [ ] 재현 단계 (Given-When-Then 형식)
- [ ] 기대 결과 vs 실제 결과
- [ ] 담당 에이전트: FE / BE
- [ ] 연관 TC 번호
- [ ] 스크린샷/로그 (가능한 경우)

### 3. 회귀 테스트 결과
- [ ] 회귀 테스트 스위트 실행 결과
- [ ] 스냅샷 테스트 diff (변경 있는 경우)
- [ ] 영향받은 기존 기능 목록

### 4. 품질 게이트 판정
- [ ] 각 게이트 항목별 통과/미통과
- [ ] 미통과 항목의 구체적 사유
- [ ] 출시 가부 추천 (Go / No-Go + 근거)

## 버그 수정 후 재검증 프로세스

1. 담당 에이전트가 수정 완료 → 완료 신호 발송
2. QA가 해당 버그의 TC 재실행
3. 수정 확인 + 회귀 테스트 실행 (사이드이펙트 확인)
4. 통과 시 TC 상태를 "통과"로 업데이트
5. 스냅샷 베이스라인 업데이트 (필요 시)

## 산출물 위치
- 테스트 시나리오: GitHub → **Notion** (Notion MCP) / GitLab → **Git Wiki** (`glab wiki`)
- 버그 리포트: **Git Issue** (`gh issue create` / `glab issue create`)
- 회귀 테스트 결과: **PR/MR 코멘트** (`gh pr comment` / `glab mr note`)
- 품질 게이트 판정: 리드에게 완료 신호로 보고

## 검증 기준
- 버그 리포트는 **재현 가능**해야 함 (재현 불가 버그는 환경 정보 추가 후 재제출)
- 회귀 테스트는 **기존 통과 TC가 여전히 통과**하는지 확인
- 품질 게이트 판정은 **주관 없이 측정 가능한 기준**으로만 판단
