# 회귀 테스트 스위트 관리 Skill

## 목적
기존 기능이 새 변경으로 깨지지 않도록 회귀 테스트를 관리하고 실행한다.
"되던 건 무조건 되게."

## 트리거
- 코드 변경이 있을 때마다 (PR 생성, 기능 추가/수정)

## 입력
- 코드 변경 diff (git diff)
- 기존 TC 목록 (회귀 스위트)
- 변경된 파일/함수 목록

## 절차
1. 변경 영향 분석:
   - git diff로 변경된 파일/함수 식별
   - 변경된 코드가 영향을 주는 기능 목록 도출
   - harness/regression-policy.md의 영향 분석 기준 참조
2. 관련 TC 선별:
   - 영향받는 기능의 기존 TC 선별
   - 공유 코드 변경 시 전체 회귀 스위트
3. 선별 TC 우선 실행 (빠른 피드백)
4. 전체 회귀 스위트 실행
5. 결과 확인:
   - 기존 TC 100% 통과 확인
   - 실패 TC → 버그 or 의도된 변경 판단
6. 새 기능 TC를 회귀 스위트에 등록

## 출력 형식
```markdown
## 회귀 테스트 결과
**실행일**: YYYY-MM-DD
**변경 원인**: [PR/커밋 정보]

### 영향 분석
| 변경 파일 | 영향 기능 | 관련 TC |
|-----------|-----------|---------|

### 실행 결과
| TC | 결과 | 비고 |
|----|------|------|

### 판정
- 기존 TC 통과율: [N/N] (100%)
- 회귀 발생: Yes / No
- 조치: [필요시]
```

## 품질 체크리스트
- [ ] 변경 영향 분석 완료
- [ ] 관련 TC 선별 실행 완료
- [ ] 전체 회귀 스위트 실행 완료
- [ ] 기존 TC 100% 통과 확인
- [ ] 새 TC를 회귀 스위트에 등록

## 안티패턴
- 변경 영향 분석 없이 전체 스위트만 실행 (느리고 비효율)
- 실패 TC를 "이미 알고 있는 문제"로 무시
- 새 기능 TC를 회귀 스위트에 등록하지 않음
