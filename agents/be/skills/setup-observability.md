# 관측 가능성 설정 Skill

## 목적
프로덕션에서 문제를 빠르게 발견/추적할 수 있는 관측 가능성을 내장한다.

## 트리거
- BE 구현 시작 시 (관측 가능성은 후순위가 아닌 기본 내장)

## 입력
- API 엔드포인트 목록
- 기술 스택 (Spring Boot / FastAPI / Node.js 등)

## 절차
1. requestId 전파 설정:
   - 요청 필터에서 X-Request-Id 헤더 추출 또는 UUID 생성
   - 로그 MDC/컨텍스트에 주입
   - 에러 응답에 requestId 포함
2. 구조화 로깅 설정:
   - JSON 형식 로그
   - 필수 필드: timestamp, level, requestId, message, context
   - 민감 데이터 마스킹 (개인정보, 시크릿)
3. API 메트릭 추적 포인트:
   - 요청 수, 응답 시간 (P50/P95/P99), 에러율
   - DB 쿼리 시간
4. 헬스체크 엔드포인트:
   - GET /health — 서비스 상태
   - GET /health/ready — 의존성(DB, 외부 API) 연결 상태

## 출력
- requestId 필터/미들웨어 코드
- 로깅 설정 파일
- 헬스체크 엔드포인트
- 메트릭 수집 포인트 (코드 주석)

## 품질 체크리스트
- [ ] requestId가 모든 로그에 포함
- [ ] 에러 응답에 requestId 포함
- [ ] 민감 데이터 마스킹 적용
- [ ] 헬스체크 엔드포인트 존재
- [ ] 주요 경로에 로깅 존재

## 안티패턴
- 로깅을 "나중에 추가" 하고 구현 후 잊어버림
- requestId 없이 로그만 남겨 디버깅 불가
- 민감 데이터(비밀번호, 토큰) 로그에 그대로 노출
