# 컴포넌트 상태 정의 Skill

## 목적
모든 인터랙티브 컴포넌트의 5 상태를 빠짐없이 정의한다.

## 트리거
- 새 컴포넌트/화면 설계 시
- 기존 컴포넌트 상태 검토 요청 시

## 입력
- 화면 목록 + 컴포넌트 목록

## 절차
1. 각 컴포넌트에 대해 5 상태 정의:

| 상태 | 질문 | 명시할 것 |
|------|------|-----------|
| idle | 처음 보이는 모습은? | 기본 UI, 플레이스홀더, 초기값 |
| loading | 무언가 기다리는 동안? | 스피너/스켈레톤, 비활성화할 요소, aria-busy |
| success | 성공하면? | 결과 표시, 다음 액션 안내, 토스트/배너 |
| error | 실패하면? | 에러 메시지 문구, 재시도 방법, 복구 경로 |
| empty | 데이터가 없으면? | 빈 상태 일러스트, CTA 버튼, 안내 문구 |

2. 각 상태의 트리거 (어떤 이벤트로 이 상태가 되는가)
3. 각 상태에서 사용자가 할 수 있는 액션
4. data-testid 지정

## 출력 형식
| 컴포넌트 | 상태 | 트리거 | 표시 | 사용자 액션 | data-testid |

## 품질 체크리스트
- [ ] 5 상태 모두 정의됨 (빠짐없이)
- [ ] 각 상태의 트리거가 명확
- [ ] error 상태에 재시도 방법 명시
- [ ] empty 상태에 안내/CTA 존재
- [ ] data-testid 지정 완료

## 안티패턴
- idle + success만 정의하고 나머지 "추후 결정"
- error 상태에서 "에러 표시" 만 쓰고 구체적 메시지 미정의
- loading에서 어떤 요소를 비활성화하는지 미명시
