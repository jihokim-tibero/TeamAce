# UX 명세 작성 Skill

## 목적
PM 기획서를 FE가 추측 없이 구현할 수 있는 UX 명세로 변환한다.

## 트리거
- PM 기획서/기능정의서 완료 후 UX 설계 요청

## 입력
- PM 기획서 (PRD) + 기능정의서
- 디자인 시스템 (있는 경우)

## 절차
1. 기획서의 사용자 시나리오에서 화면 목록 도출
2. 화면별 컴포넌트 식별
3. 사용성 분석 — "사용자가 고민 없이 목표 달성 가능한가?"
4. 컴포넌트 상태 정의 (5 상태 필수):
   - idle: 초기/대기 — 무엇을 표시하는가
   - loading: 로딩 중 — 스피너/스켈레톤/프로그레스
   - success: 성공 — 결과 표시 + 다음 액션
   - error: 에러 — 에러 메시지 + 재시도 방법
   - empty: 데이터 없음 — 빈 상태 UI + 안내
5. data-testid 네이밍 — [컴포넌트]-[역할] 형식
6. 디자인 토큰 정의 (색상/타이포/간격/반경)
7. 사용자 여정 플로우 작성
8. 인터랙션 명세 (스크롤, 포커스, 트랜지션)
9. 접근성 기준 (WCAG AA)
10. FE 구현 노트 + QA 검증 포인트

## 출력 형식
agents/ux.md의 UX 명세 문서 형식 참조

## 품질 체크리스트
- [ ] 기능정의서 기능 대비 화면 매핑 100%
- [ ] 모든 인터랙티브 컴포넌트에 5 상태 정의
- [ ] 모든 인터랙티브 요소에 data-testid
- [ ] 디자인 토큰 (색상/타이포/간격/반경) 정의
- [ ] 접근성 섹션 존재
- [ ] 필드명이 PM 기능정의서와 일치
- [ ] "적절히 처리" 같은 모호 표현 없음

## 안티패턴
- 상태를 3개만 정의하고 error/empty 누락
- data-testid 없이 "버튼" 같은 일반명사로 지칭
- FE가 추측해야 하는 미정 항목 남김
