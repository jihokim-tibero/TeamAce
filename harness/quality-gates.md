# 품질 게이트 정의

## 목적

각 Phase 완료 시 **측정 가능한 기준**으로 검증하여, 미달 시 다음 Phase 진행을 차단한다.
리드 에이전트가 이 기준을 적용하여 자동 판정한다.

---

## Phase 1: 기획 완료 게이트

**담당**: PM Agent
**검증자**: Lead Agent

| # | 항목 | 기준 | 검증 방법 |
|---|------|------|-----------|
| 1.1 | 기획서 필수 섹션 | 배경/목적, 목표지표, 범위, 사용자시나리오, 의존성, Agent인터페이스 모두 존재 | 섹션 헤딩 존재 여부 |
| 1.2 | 목표 지표 측정 가능성 | 모든 KPI에 수치 또는 비율 기준 포함 | "~% 이상", "~초 이내" 등 정량 표현 존재 |
| 1.3 | 기능정의서 완성도 | 모든 기능에 입력/처리/출력/예외 4항목 정의 | 기능별 4항목 존재 여부 |
| 1.4 | 수락 기준 검증 가능성 | Given-When-Then 변환 가능한 수준 | 모호 표현("적절히", "필요 시") 부재 |
| 1.5 | 에이전트 인터페이스 | UX/FE/BE/QA 각각에 대한 지시 존재 | 섹션 존재 + 내용 비어있지 않음 |
| 1.6 | 필드 네이밍 | 기능정의서 내 데이터 필드명이 camelCase | 네이밍 규약 준수 |

**통과 기준**: 6/6 항목 충족

## Phase 4-fix: 수정 루프 절차

QA가 No-Go 판정을 내린 경우 Lead가 다음 절차를 실행합니다.

### 루프 절차

| 단계 | 담당 | 행동 |
|------|------|------|
| 1. 이슈 분류 | Lead | QA Git Issue를 읽고 FE/BE 담당 배정 |
| 2. 수정 지시 | Lead → FE/BE | Issue URL + 수정 요청 전달. 같은 `feature/[feature-name]` 브랜치에서 수정 |
| 3. 수정 구현 | FE/BE | 코드 수정 + 자체 품질 게이트(Phase 3) 재검증 |
| 4. 수정 완료 | FE/BE → Lead | `[FE FIX DONE]` / `[BE FIX DONE]` 신호 |
| 5. 재검증 | QA | 해당 TC 재실행 + 회귀 테스트 실행 |
| 6. 판정 | QA → Lead | Go → 완료 / No-Go → 2단계로 복귀 |

### 루프 제한

- **최대 3회 반복** — 3회 후에도 미통과 시 사용자에게 현황 보고 + 판단 요청
- 매 루프마다 미해결 Issue 수가 감소해야 함 (감소하지 않으면 접근 방식 재검토)
- Critical 버그는 반드시 해당 루프 내에서 해결 (다음 루프로 이월 금지)
