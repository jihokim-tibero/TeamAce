# 회귀 방지 정책

## 핵심 원칙

**"되던 건 무조건 되게 한다."**

새로운 변경이 기존에 정상 작동하던 기능을 깨뜨리는 것을 허용하지 않는다.

---

## 1. 회귀 테스트 스위트

### 구성
- 모든 기능의 Critical/High TC를 회귀 테스트 스위트에 등록
- 새 기능 출시 시 해당 기능의 TC가 자동으로 회귀 스위트에 추가
- 회귀 스위트는 **누적** — 기능이 제거되지 않는 한 TC를 삭제하지 않음

### 실행 시점
- 모든 PR 생성 시 (CI 파이프라인 통합)
- Phase 4 (검증) 진입 시
- 배포 전 최종 검증 시

### 실패 시 대응
1. 실패한 TC 목록 + 원인 분석
2. 해당 코드 변경을 담당한 에이전트에 수정 요청
3. 수정 후 전체 회귀 스위트 재실행
4. 100% 통과 전까지 머지/배포 차단

---

## 2. 변경 영향 분석

### 프로세스
1. 코드 변경 diff 분석
2. 변경된 파일/함수가 영향을 주는 기능 식별
3. 해당 기능의 TC만 선별적으로 우선 실행 (빠른 피드백)
4. 이상 없으면 전체 회귀 스위트 실행

### 영향 분석 기준

| 변경 유형 | 영향 범위 | 실행할 TC |
|-----------|-----------|-----------|
| FE 컴포넌트 수정 | 해당 화면 + 부모 화면 | 해당 화면 TC + 네비게이션 TC |
| BE API 수정 | 해당 API 호출자 전체 | API TC + 호출하는 FE 화면 TC |
| DB 스키마 변경 | 해당 테이블 사용 전체 | 관련 API TC + 마이그레이션 TC |
| 공유 컴포넌트 수정 | 사용하는 모든 화면 | 전체 회귀 스위트 |
| 설정/환경 변경 | 전체 | 전체 회귀 스위트 |

---

## 3. 스냅샷 테스트

### UI 스냅샷
- 주요 화면의 각 상태(idle/loading/success/error/empty)를 스냅샷으로 저장
- 코드 변경 후 스냅샷 비교 → 의도하지 않은 UI 변경 감지
- **의도된 변경**: 스냅샷 베이스라인 업데이트 (리드 승인 필요)
- **의도하지 않은 변경**: 버그로 분류, 수정 요청

### API 응답 스냅샷
- 주요 API 엔드포인트의 성공/에러 응답을 스냅샷으로 저장
- 응답 구조(필드명, 타입) 변경 감지
- 필드 추가는 허용, 필드 삭제/타입 변경은 차단 (하위 호환성)

### 스냅샷 관리
- 스냅샷 파일은 Git에서 버전 관리
- 스냅샷 업데이트는 별도 커밋으로 분리 (변경 이유 기록)
- 스냅샷 diff가 있으면 PR 리뷰에서 반드시 확인

---

## 4. 사이드이펙트 최소화 원칙

### 코드 변경 시
- 변경 범위를 최소화 — 필요한 부분만 수정
- 공유 코드 수정 시 영향 범위를 사전 분석
- 리팩터링과 기능 변경을 같은 PR에 섞지 않음

### API 변경 시
- 하위 호환성 유지 (필드 추가 OK, 삭제/변경 NO)
- 비호환 변경이 필요하면 새 버전 엔드포인트 생성 (v1 → v2)
- deprecated 필드는 최소 1 릴리즈 동안 유지

### DB 변경 시
- 컬럼 추가는 NOT NULL 기본값 포함
- 컬럼 삭제는 2단계: 1) 코드에서 사용 제거 → 2) 다음 릴리즈에서 컬럼 삭제
- 마이그레이션은 롤백 가능하도록 작성
