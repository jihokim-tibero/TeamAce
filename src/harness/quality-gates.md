# 품질 게이트 정의

## 목적

각 Phase 완료 시 **측정 가능한 기준**으로 검증하여, 미달 시 다음 Phase 진행을 차단한다.
리드 에이전트가 이 기준을 적용하여 자동 판정한다.

---

## Phase 1: 기획 완료 게이트

**담당**: PM Agent
**검증자**: Lead Agent

| # | 항목 | 기준 | 검증 방법 |
|---|------|------|-----------|
| 1.1 | 기획서 필수 섹션 | 배경/목적, 목표지표, 범위, 사용자시나리오, 의존성, Agent인터페이스 모두 존재 | 섹션 헤딩 존재 여부 |
| 1.2 | 목표 지표 측정 가능성 | 모든 KPI에 수치 또는 비율 기준 포함 | "~% 이상", "~초 이내" 등 정량 표현 존재 |
| 1.3 | 기능 명세서 완성도 | 모든 기능에 입력/처리/출력/예외 4항목 정의 | 기능별 4항목 존재 여부 |
| 1.4 | 수락 기준 검증 가능성 | Given-When-Then 변환 가능한 수준 | 모호 표현("적절히", "필요 시") 부재 |
| 1.5 | 에이전트 인터페이스 | PUB/FE/BE/QA 각각에 대한 지시 존재 | 섹션 존재 + 내용 비어있지 않음 |
| 1.6 | 필드 네이밍 | 기능 명세서 내 데이터 필드명이 camelCase | 네이밍 규약 준수 |

## Phase 4: 검증 완료 게이트

**담당**: QA Agent
**검증자**: Lead Agent

| # | 항목 | 기준 | 검증 방법 |
|---|------|------|-----------|
| 4.1 | Critical TC | 100% 통과 | QA 리포트 |
| 4.2 | High TC | ≥ 95% 통과 | QA 리포트 |
| 4.3 | E2E 테스트 | Critical Path 100% 통과 | Playwright 테스트 결과 |
| 4.4 | 회귀 테스트 | 기존 TC 100% 통과 | 회귀 스위트 결과 |
| 4.5 | API 응답 시간 | P95 < 500ms | 성능 테스트 결과 |
| 4.6 | 접근성 | WCAG AA 통과 | 자동화 검사 결과 |
| 4.7 | 에러율 | < 0.1% | 테스트 중 에러 발생률 |
| 4.8 | 블랙박스 원칙 준수 | TC가 명세(PM·PUB·BE 명세)에서 도출되고 소스 내부에 의존하지 않음. 선택자는 data-testid·공개 API 스키마만 사용 | TC 본문·선택자 검사 |
| 4.9 | 명세 불일치 처리 | 명세와 동작이 다른 항목은 TC 임의 수정 없이 버그/명세 정정 이슈로 등록되어 있음 | Git Issue 확인 |

## Phase 4-fix: 수정 루프 절차

QA가 No-Go 판정을 내린 경우 Lead가 다음 절차를 실행합니다.

### 루프 절차

| 단계 | 담당 | 행동 |
|------|------|------|
| 1. 이슈 분류 | Lead | QA Git Issue를 읽고 PUB/FE/BE 담당 배정 |
| 2. 수정 지시 | Lead → PUB/FE/BE | Issue URL + 수정 요청 전달. 같은 `feature/[feature-name]` 브랜치에서 수정 |
| 3. 수정 구현 | PUB/FE/BE | 코드 수정 + 자체 품질 게이트(Phase 2/3) 재검증 |
| 4. 수정 완료 | PUB/FE/BE → Lead | `[PUB FIX DONE]` / `[FE FIX DONE]` / `[BE FIX DONE]` 신호 |
| 5. 재검증 | QA | 해당 TC 재실행 + 회귀 테스트 실행 |
| 6. 판정 | QA → Lead | Go → 완료 / No-Go → 2단계로 복귀 |

### 루프 제한

- **최대 3회 반복** — 3회 후에도 미통과 시 사용자에게 현황 보고 + 판단 요청
- 매 루프마다 미해결 Issue 수가 감소해야 함 (감소하지 않으면 접근 방법 재검토)
