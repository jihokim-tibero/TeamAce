# PM → PUB 핸드오프 계약

## PM이 PUB에게 전달해야 하는 필수 산출물

### 1. 기획서 (PRD)
- [ ] 배경 및 목적 (왜 이 기능이 필요한가)
- [ ] 타깃 사용자 정의
- [ ] 사용자 시나리오 (주요 흐름 + 예외 흐름)
- [ ] 목표 지표 (측정 가능한 KPI)
- [ ] 범위 (In Scope / Out of Scope)

### 2. 기능 명세서
- [ ] 기능 목록 테이블 (기능ID, 기능명, 우선순위)
- [ ] 각 기능의 입력 조건 / 처리 로직 / 출력 결과 / 예외 처리
- [ ] 상태 전이가 있는 기능은 상태 목록 명시

### 3. PUB 에이전트를 위한 지시
- [ ] 필요한 화면 목록 (또는 "PUB 판단에 위임" 명시)
- [ ] 특별한 디자인 제약사항 (브랜드 가이드, 기존 디자인 시스템 등)
- [ ] 디자인 톤/분위기 참고 (있는 경우)
- [ ] 참고할 경쟁사/레퍼런스 (있는 경우)

## 필드 네이밍
- 기능 명세서에 정의된 필드명이 이후 전체 파이프라인의 기준
- PUB는 이 필드명을 data-testid, 컴포넌트 Props에 그대로 사용

## 산출물 위치
- 기획서 (PRD): GitHub → **Notion** (Notion MCP) / GitLab → **Git Wiki** (`glab api`)
- 기능 명세서: GitHub → **Notion** (Notion MCP) / GitLab → **Git Wiki** (`glab api`)
- PUB Agent는 Notion 또는 Git Wiki에서 기획서·기능 명세서를 조회

## 검증 기준
- PUB Agent가 Notion/Wiki의 기획서만 보고 "어떤 화면이 필요한지, 각 화면에서 무슨 일이 일어나는지" 파악 가능해야 함
- 모호한 표현 금지: "적절히 처리", "필요 시 표시" 등은 불합격
