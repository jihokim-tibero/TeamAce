# QA 핵심 설계 원칙

작업 시작 전 읽고, 모든 작업 과정에서 준수한다. 완료 시 산출물 대상으로 재확인한다.

---

## 원칙

### P0. 플랜 우선
작업 시작 전 **플랜 모드로 전환하여 작업 계획을 수립**한 후 실행한다.
계획 없이 바로 구현에 착수하면 실패. 플랜에는 작업 범위, 산출물 목록, 주요 판단 근거를 포함한다.

### P1. 리스크 기반 우선순위
모든 TC에 비즈니스 임팩트 기반의 우선순위(Critical/High/Medium/Low)가 있어야 한다.
우선순위 근거가 명시되지 않은 TC가 있으면 실패.

### P2. 회귀 허용 불가
기존 기능의 TC가 새 변경으로 인해 실패하면 안 된다.
회귀 테스트 스위트 100% 통과가 필수. 기존 TC 실패 시 실패.

### P3. Given-When-Then 완전성
모든 TC가 Given(초기 상태)-When(액션)-Then(기대 결과) 형식이어야 한다.
형식이 불완전하거나 모호한 TC가 있으면 실패.

### P4. data-testid 기반 자동화
E2E 테스트는 data-testid로만 요소를 선택한다.
CSS selector, XPath 등 다른 선택자를 사용하면 실패.

### P5. 재현 가능한 버그 리포트
모든 버그 리포트는 제3자가 재현 가능해야 한다.
재현 단계가 불명확하거나 환경 정보가 누락된 리포트가 있으면 실패.

### P6. Critical TC 100% 통과
Critical 우선순위 TC는 하나도 실패할 수 없다.
Critical TC 실패가 있으면 Go 판정 불가.
