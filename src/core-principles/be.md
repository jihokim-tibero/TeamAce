# BE 핵심 설계 원칙

작업 시작 전 읽고, 모든 작업 과정에서 준수한다. 완료 시 산출물 대상으로 재확인한다.

---

## 원칙

### P0. 플랜 우선
작업 시작 전 **플랜 모드로 전환하여 작업 계획을 수립**한 후 실행한다.
계획 없이 바로 구현에 착수하면 실패. 플랜에는 작업 범위, 산출물 목록, 주요 판단 근거를 포함한다.

### P1. 객체지향 패러다임 준수
객체지향 원칙(캡슐화, 상속, 다형성, 추상화)을 준수한다.
절차적으로 나열된 로직, God 클래스, 책임이 불명확한 클래스가 있으면 실패.

### P2. 네이밍 규약 엄수
- 모든 변수·메서드: camelCase
- 모든 상수: UPPER_SNAKE_CASE
- 리터럴 직접 사용 금지. 코드 파일 상단에 상수로 선언 후 사용.
매직 넘버/매직 스트링이 하나라도 있으면 실패.

### P3. 추상화 극대화
공통 코드, 비슷한 구조는 최대한 추상화한다.
동일한 로직이 2곳 이상 중복되면 실패. 공통 인터페이스/추상 클래스로 추출한다.

### P4. 도메인 명확 정의 + 상호 참조 금지
도메인을 명확히 정의하고, 도메인 계층 간 상호 참조를 금지한다.
도메인 A가 도메인 B를 직접 import하면 실패.
두 개 이상 도메인에 걸치는 로직은 반드시 서비스 레이어에서 작성한다.

### P5. 계층 구조 엄수: api → service → domain → infrastructure
- api 계층: 함수 당 하나의 도메인만 다루는 경우 직접 호출 허용. 두 개 이상 도메인 조합 시 반드시 서비스 계층 경유.
- service 계층: 비즈니스 로직, 도메인 간 조합.
- domain 계층: 엔티티, 값 객체, 도메인 로직.
- infrastructure 계층: DB, 외부 API, 파일 시스템.
하위 계층이 상위 계층을 import하면 실패.

### P6. 경계 검증 필수
모든 외부 입력(API 요청, DB 조회 결과, 외부 API 응답)에 유효성 검증이 있어야 한다.
검증 없이 외부 데이터를 신뢰하면 실패.

### P7. 에러 코드 + requestId 필수
모든 에러 응답에 에러 코드와 requestId가 포함되어야 한다.
에러 코드 없는 에러 응답이 있으면 실패.

### P8. 구조화 로깅
console.log 금지. 모든 주요 경로에 구조화된 로그(JSON + requestId)가 있어야 한다.
로깅 프레임워크 변경 시 컨텍스트 API가 달라질 수 있다 (예: Logback MDC ↔ Log4j2 ThreadContext). 프레임워크에 맞는 API를 사용하지 않으면 requestId가 로그에 누락된다.

### P9. 하위 호환성
필드 추가는 OK. 필드 삭제/타입 변경은 API 버전 업 필수.
기존 클라이언트가 깨질 수 있는 변경이 버전 업 없이 이루어지면 실패.

### P10. 화이트박스 단위 테스트 — 커버리지 ≥ 80%
BE는 **소스 코드 내부 구조를 아는 화이트박스 관점**에서 **단위 테스트**를 작성한다.
- 대상: service·domain 로직, 유효성 검증, 에러 분기, 경계 조건. 레이어별 단위로 검증한다.
- 커버리지: ≥ 80%. 외부 의존성(DB, 외부 API)은 모킹하여 로직만 독립 검증한다.
- 외부 계약 관점의 E2E·API 블랙박스 테스트는 **QA 책임** — BE가 작성하지 않는다.
- 하드코딩된 시크릿이 있으면 실패.
미달 시 실패.

### P11. 관측 가능성
- 모든 외부 API 호출에 타임아웃 설정
- P95 응답시간 < 500ms 목표
- 신규 API에 문서화 없이 배포 금지
