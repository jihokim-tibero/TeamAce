# FE 핵심 설계 원칙

작업 시작 전 읽고, 모든 작업 과정에서 준수한다. 완료 시 산출물 대상으로 재확인한다.

---

## 원칙

### P0. 플랜 우선
작업 시작 전 **플랜 모드로 전환하여 작업 계획을 수립**한 후 실행한다.
계획 없이 바로 구현에 착수하면 실패. 플랜에는 작업 범위, 산출물 목록, 주요 판단 근거를 포함한다.

### P1. any 타입 제로
코드 전체에 `any` 타입이 하나도 없어야 한다.
`grep -r ": any"` 결과가 0건이어야 한다. 위반 시 실패.

### P2. PUB 영역 불가침
PUB 소유 파일(views/, shared/ui/, tokens.css)을 수정하지 않는다.
`git diff`에 PUB 파일 변경이 있으면 실패.

### P3. 데이터 변환 순수성
모든 데이터 트랜스폼 함수는 순수 함수여야 한다.
side-effect(API 호출, DOM 조작, 전역 상태 변경)가 있으면 실패.

### P4. ErrorBoundary 전면 적용
모든 페이지와 주요 섹션에 ErrorBoundary가 래핑되어 있어야 한다.
미적용 페이지가 있으면 실패.

### P5. 테스트 커버리지 ≥ 80%
전체 테스트 커버리지가 80% 이상이어야 한다.
데이터 트랜스폼 함수는 90% 이상.

### P6. 빌드 성공
`npm run build`가 에러 없이 성공해야 한다.
컴파일 에러, 린트 에러가 있으면 실패.

### P7. 코드 위생
- 파일당 300줄 이하, 함수당 50줄 이하
- API 호출은 반드시 hooks를 통해서만 (컴포넌트에서 직접 fetch 금지)
- 테스트 없는 hooks 제출 금지

### P8. 중복 코드 금지 — 공통화·추상화 필수
동일한 로직·패턴이 2곳 이상에서 반복되면 반드시 공통 모듈(hooks, utils, helpers)로 추상화한다.
코드 복사-붙여넣기로 인한 중복이 발견되면 실패.

### P9. 접근성 구현 세부 원칙
- 로딩/비활성 상태의 버튼·컨테이너에 `aria-busy` 속성 부여
- 폼 레이블에 `htmlFor`과 `id` 쌍을 반드시 명시 (시각적 래핑과 무관)
- 동일 역할의 버튼이 여러 상태에 걸쳐 존재하면 `data-testid`를 통일하고, 부모 요소의 `aria` 속성으로 맥락 구분
위반 시 접근성 감사 불합격.

### P10. 디자인 시스템 기반 개발
PUB이 구축한 디자인 시스템(`shared/ui/`, `styles/tokens.css`)을 반드시 활용한다.
반복되는 UI 구성은 공통 컴포넌트를 조합하여 구현하고, **각 페이지는 입력 데이터(Props)만 변경**되도록 설계한다.
페이지별로 동일 패턴의 컴포넌트를 새로 만들지 않는다 — 기존 공통 컴포넌트를 재사용하고, 없으면 PUB에 공통 컴포넌트 추가를 요청한다.
