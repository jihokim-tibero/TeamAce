# 기획서 작성 Skill

## 목적
사용자 요구사항을 제품 방향과 비전을 설명하는 기획서로 변환한다. 도메인별 사용자 기능 리스트를 포함하여 후속 에이전트가 전체 그림을 이해할 수 있게 한다.

## 트리거
- 새 기능/서비스 기획 요청
- 기존 기능 개선 기획 요청

## 입력
- 사용자 요구사항 (구두 또는 문서)
- 기존 데이터/지표 (있는 경우)
- 경쟁사/시장 정보 (있는 경우)

## 절차
1. 요구사항 명확화 — 모호한 부분 질문으로 해소
2. 제품 비전 수립 — 이 제품/서비스가 추구하는 방향과 궁극적 목표
3. 가설 수립 — "~하면 ~할 것이다" 형태
4. 배경/목적 작성 — 데이터 근거 포함
5. 목표 지표(KPI) 정의 — 현재값 → 목표값, 측정 방법
6. 범위 정의 — In Scope / Out of Scope
7. 타깃 사용자 정의 — 세그먼트 + 핵심 니즈
8. **도메인별 기능 리스트 작성** — 기능을 도메인으로 분류하고 리스트업
   - 각 도메인의 목적과 사용자 가치 설명
   - 기능마다 ID, 이름, 설명, 우선순위 부여
9. 사용자 시나리오 — 주요 흐름 + 예외 흐름 (최소 2개)
10. 의존성 파악
11. Agent 인터페이스 작성 — PUB/FE/BE/QA 각각에 구체적 지시
12. 성공 판정 기준 정의

## 출력 형식
`~/.claude/agents/pm.md`의 기획서 형식 참조

## 기획서 ↔ 기능 명세서 관계
기획서의 **도메인별 기능 리스트**가 기능 명세서의 입력이 된다.
기획서를 먼저 완성한 후, 리스트의 각 항목에 대해 write-spec 스킬로 기능 명세서를 작성한다.

## 품질 체크리스트
- [ ] 제품 비전이 명확하게 기술되어 있는가
- [ ] 가설이 검증 가능한 형태인가
- [ ] 모든 KPI에 수치 기준이 있는가
- [ ] In/Out Scope가 명확한가
- [ ] 도메인별 기능 리스트가 빠짐없이 작성되었는가
- [ ] 각 기능에 ID와 우선순위가 부여되었는가
- [ ] 사용자 시나리오에 예외 흐름이 포함되었는가
- [ ] Agent 인터페이스 4개 에이전트 모두 작성했는가
- [ ] 모호 표현("적절히", "필요 시")이 없는가

## 안티패턴
- 데이터/근거 없이 직감으로 기획
- "~하면 좋겠다" 식의 비검증 가능한 목표
- 기능을 도메인 없이 flat list로 나열
- Agent 인터페이스 섹션 누락
- 비전 없이 기능만 나열
