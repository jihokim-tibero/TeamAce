# 기능 명세서 작성 Skill

## 목적
기획서의 도메인별 기능 리스트 각 항목에 대해 깊은 뎁스의 상세 명세를 작성한다.
개발 에이전트(PUB/FE/BE)가 추측 없이 구현할 수 있는 수준의 명세를 목표로 한다.

## 트리거
- 기획서 작성 완료 후
- 기존 기능 변경/추가 시

## 입력
- 완성된 기획서 (도메인별 기능 리스트 포함)

## 절차
1. 기획서의 **도메인별 기능 리스트**에서 각 기능 항목 추출
2. 기능별 상세 명세 작성:
   - 기능 개요 (사용자 가치와 동작 요약)
   - 입력 조건 (사전 조건 + 입력 데이터 필드명 camelCase + 유효성 검증)
   - 처리 로직 (비즈니스 규칙, 분기 조건 명시, 외부 시스템 호출)
   - 출력 결과 (데이터 필드명 camelCase + 사용자 피드백)
   - 예외 처리 (에러 코드 + 발생 조건 + 사용자 표시 + 복구 경로)
   - 상태 전이 (해당 시)
3. 수락 기준 작성 — Given-When-Then 형태 (기능당 최소 2개)
4. 관련 기능 의존 관계 명시
5. ~/.claude/teamace/contracts/pm-to-pub.md, pm-to-be.md 체크리스트 대조

## 출력 형식
`~/.claude/agents/pm.md`의 기능 명세서 형식 참조

## 기능 명세서 작성 단위
- 기획서의 기능 리스트에 있는 **기능ID 하나당 기능 명세서 하나**
- 여러 기능을 하나의 문서에 묶을 수도 있고, 기능별로 분리할 수도 있음
- 단, 각 기능의 명세 뎁스는 동일해야 함

## 품질 체크리스트
- [ ] 기획서의 기능 리스트 대비 명세 매핑률 100%
- [ ] 기능당 입력/처리/출력/예외 4항목 모두 정의
- [ ] 입력 데이터 필드마다 타입, 필수 여부, 유효성 검증 명시
- [ ] 처리 로직의 분기 조건이 if/then/else 수준으로 구체적
- [ ] 에러 코드가 구체적으로 정의되고 복구 경로 포함
- [ ] 필드명 camelCase 준수
- [ ] 수락 기준이 Given-When-Then 형태이고 기능당 최소 2개
- [ ] 관련 기능 의존 관계가 명시됨

## 안티패턴
- "처리한다"만 쓰고 구체적 로직 미명시
- 필드명 snake_case 또는 한글 혼용
- 에러 처리 항목 누락
- 유효성 검증 규칙 미정의
- 기획서 기능 리스트에 있는 항목을 빠뜨림
